# 用紙サイズ未反映 調査メモ（2026-03-18）

## 1. 依頼内容
- 事象: 印刷設定のうち「用紙サイズ変更」だけ反映されない
- 条件: SumatraPDF を使用
- 本ドキュメントでは原因追求のみ実施し、コード修正は未実施

## 2. 今回の調査範囲（読んだ箇所）
- UI送信
  - `components/PdfViewer.tsx` の `handleServerPrint`
  - `paperSize: printPaperSize || undefined` を `/api/print` へPOST
- API正規化
  - `app/api/print/route.ts` の `normalizePaperSize`
- Sumatra実行
  - `app/api/print/route.ts` の `printWindows`
  - `-print-settings` 組み立て: `color/duplex/paper`
  - 実行: `runSumatraPrint()`
- 補助API
  - `app/api/print/printers/route.ts`（プリンター名一覧のみ）
- 公式仕様確認
  - SumatraPDF Command Line Docs
  - `-print-settings` の `paper=<page size>` は有効仕様（A2,A3,A4,A5,A6,letter,legal,tabloid,statement）

## 3. 事実ベースの観測
1. UIから `paperSize` は送信されている
   - `printPaperSize` が空でなければ API に渡る実装
2. APIでの正規化は実施済み
   - 許可値: `A3/A4/A5/B4/B5/Letter/Legal`（内部的に Letter/Legal は lower-case 化）
3. Sumatra呼び出し時に `paper=<size>` は `-print-settings` に含める実装
4. ただし、APIで受け入れているサイズと Sumatra公式の記載にズレがある
   - APIは `B4/B5` を許可
   - 公式の印刷オプション記載には `B4/B5` が明示されていない
5. `-print-settings` には `fit/shrink/noscale` を付けていない
   - 用紙サイズは指定しているが、縮尺方針は明示していない
6. プリンターがその用紙サイズをサポートするかの事前検証がない
   - `/api/print/printers` はプリンター名だけ返す
   - 用紙対応可否は未取得

## 4. 原因候補（優先度つき）

### A. プリンタードライバー側で `paper=<size>` が無視・置換される（最有力）
- 根拠:
  - 紙サイズ指定はアプリ側が送っても、ドライバー非対応時は既定紙へフォールバックされることがある
  - 現状コードは「そのプリンターが指定紙に対応しているか」を事前確認していない
- 典型症状:
  - カラー/両面は効くが、紙サイズだけ既定のまま

### B. `-print-settings` に `fit/shrink` がなく、体感上「サイズが変わっていない」ように見える（有力）
- 根拠:
  - Sumatra仕様上、縮尺は別指定（`fit`, `shrink`, `noscale`）
  - 例えば A4 PDF を A3 指定しても、拡大方針がなければ期待した見え方にならない
- 典型症状:
  - 用紙は変わっているが、印字サイズが想定と異なり「未反映」に見える

### C. API許可値と Sumatra公式値の不一致（B4/B5）が無言失敗を誘発（有力）
- 根拠:
  - APIは `B4/B5` を受け付ける
  - 公式記載の `paper=<size>` 例には B 系が見当たらない
  - `-silent` 付き実行だと利用者に気づきにくい
- 典型症状:
  - A系は効くが B系だけ効かない（または既定に戻る）

### D. UI上の「用紙サイズ変換」と「用紙サイズ（印刷）」の混同（中）
- 根拠:
  - 同じ設定パネル内に2系統のサイズ設定がある
  - 一方はPDF変換、他方は印刷設定
- 典型症状:
  - 表示は変わるが印刷紙が変わらない、またはその逆

### E. 実際は Sumatra 経路ではなく別経路で印刷されている（中）
- 根拠:
  - 実装は Sumatra 優先だが、環境差で検出失敗すると Adobe/Foxit 経路へ落ちる
  - その場合、紙サイズ適用ロジックは別実装に依存
- 典型症状:
  - サーバーログに `backend=sumatra` が出ない

## 6. 実施した修正

### 修正1: `normalizePaperSize` での大文字小文字統一と B4/B5 除外
**ファイル**: `app/api/print/route.ts`

**変更内容**:
- すべてのサイズを小文字に統一（Sumatra と PowerShell が小文字を期待）
- B4/B5 を除外（Sumatra 公式仕様に非対応）
- 対応値を Sumatra 公式ドキュメント準拠に限定

**変更前**:
```typescript
const upper = value.toUpperCase();
if (["A3", "A4", "A5", "B4", "B5"].includes(upper)) return upper;
if (upper === "LETTER") return "letter";
if (upper === "LEGAL") return "legal";
```

**変更後**:
```typescript
// Sumatra と PowerShell が認識できる値に統一（すべて小文字）
// SumatraPDF 公式サポート値: a3, a4, a5, a6, letter, legal, tabloid, statement
// B4/B5 は Sumatra 非対応のため除外
if (upper === "A3") return "a3";
if (upper === "A4") return "a4";
if (upper === "A5") return "a5";
if (upper === "A6") return "a6";
if (upper === "LETTER") return "letter";
if (upper === "LEGAL") return "legal";
if (upper === "TABLOID") return "tabloid";
if (upper === "STATEMENT") return "statement";
```

### 修正2: Adobe/Foxit 経路での PowerShell 値マッピング
**ファイル**: `app/api/print/route.ts`

**変更内容**:
- Sumatra 内部値（小文字）を PowerShell 期待値（大文字または固有名）にマップ
- PowerShell の `Set-PrintConfiguration -PaperSize` が認識できる形式に変換

**変更箇所**: `applyPrinterConfiguration` 関数内で、以下を追加:
```typescript
// PowerShell の Set-PrintConfiguration に渡す紙サイズを変換
let powerShellPaperSize = "";
if (paperSize) {
  const psMap: Record<string, string> = {
    "a3": "A3",
    "a4": "A4",
    "a5": "A5",
    "a6": "A6",
    "letter": "Letter",
    "legal": "Legal",
    "tabloid": "Tabloid",
    "statement": "Statement",
  };
  powerShellPaperSize = psMap[paperSize.toLowerCase()] || "";
}
```
